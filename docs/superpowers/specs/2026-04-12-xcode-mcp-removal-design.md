# Xcode MCP Removal Design

## Goal

Remove all checked-in references to Xcode MCP / XcodeBuildMCP from this repository, including generated comments and any local templates or scaffold paths that would reintroduce that wording later.

## Scope

- Remove contributor-facing guidance that explicitly requires XcodeBuildMCP.
- Remove historical scaffold attribution text from docs.
- Remove generated comment headers in checked-in iOS config files.
- Search for any generator, template, or scaffold source in this repo that still emits the wording and update it if present.
- Keep runtime behavior, build settings, and project structure unchanged.

## Non-Goals

- Replace Xcode MCP with a new tool integration.
- Rework the broader Swift contributor workflow beyond removing the specific MCP dependency wording.
- Modify user-owned unrelated changes already present in the worktree.

## Current Footprint

Current checked-in references are:

- `swift/CLAUDE.md`
- `swift/README.md`
- `examples/ios/Config/Debug.xcconfig`
- `examples/ios/Config/Release.xcconfig`
- `examples/ios/Config/Shared.xcconfig`
- `examples/ios/Config/Tests.xcconfig`

If a generator or template source exists in-repo, it must also be updated so the wording does not reappear.

## Approach

1. Search the repository for `XcodeBuildMCP`, `xcodebuildmcp`, `xcode mcp`, and `Generated by XcodeBuildMCP`.
2. Update documentation text to remove tool-specific MCP references while preserving any still-useful project guidance.
3. Update checked-in config comments to remove scaffold-tool attribution without changing actual config values.
4. Re-run repository-wide search to confirm the strings are fully removed.

## Design Decisions

### Documentation strategy

Where a section is entirely about XcodeBuildMCP usage, remove the section rather than replacing it with inaccurate generic guidance. Where a sentence only uses XcodeBuildMCP as attribution, rewrite or delete that sentence.

### Generated file strategy

Because the xcconfig files are committed source, their comments should be cleaned directly. If there is no in-repo generator/template path for those comments, direct cleanup of the checked-in files is sufficient.

### Safety

Only text and comments should change. No build settings, source code behavior, or package definitions should be modified as part of this task.

## Verification

- Repository-wide search returns no matches for:
  - `XcodeBuildMCP`
  - `xcodebuildmcp`
  - `xcode mcp`
  - `Generated by XcodeBuildMCP`
- Edited docs still read coherently.
- The iOS config files keep the same settings after comment cleanup.

## Risks

- A reference could exist in an overlooked generated artifact or documentation file.
- Removing an entire documentation section could drop guidance that should instead be rewritten more generally.

Mitigation: inspect each match individually and preserve any tool-agnostic guidance that remains valuable.
